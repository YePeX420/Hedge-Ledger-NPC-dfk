You are a senior backend engineer working inside an existing Node.js (ESM) webapp codebase that uses:
- Express (web server + REST endpoints)
- Postgres via `pg` for SQL execution
- Drizzle ORM for schema definitions + migrations

Goal: Fix the Hero Price Tool so “sales” are canonical, reliable, and scalable.
Right now `tavern_sales` is polluted because it treats “disappeared listing” as a sale and uses last ask price.
Implement a production-grade pipeline that:
1) Tracks open tavern auctions reliably
2) Finalizes closed auctions by querying `purchasePrice` (true paid price)
3) Distinguishes SOLD vs DELISTED
4) Makes pricing comps use only SOLD rows (canonical), with listing-only fallback clearly labeled

IMPORTANT DFK GRAPHQL RULES (MUST FOLLOW):
- Do NOT use `where: { open: true }` to represent “currently open” listings. Some auctions can be stuck open even when completed.
  Use: `where: { endedAt: null, winner: null }` (also excludes private auctions).
- A sale is only confirmed if `purchasePrice` is non-null on the auction.
- Use `endedAt` as the sale timestamp when SOLD.
- Avoid N+1 hero fetches: pull hero fields through `tokenId { ... }` in the same auctions query.

C) INGESTION PIPELINE (WEBAPP-SAFE + HORIZONTALLY SCALABLE)
--------------------------------------------------------------------
Do NOT rely on in-memory state between runs (webapps scale horizontally).
Persist job state in DB and use Postgres advisory locks so only one worker runs a job per realm at a time.

Implement the ingestion as a separate “worker” module that can be run in one of these modes:
- Mode 1 (Recommended): separate Node process (e.g., `node worker.js`) started by the platform scheduler/cron
- Mode 2: in-app scheduler (setInterval) BUT still safe under horizontal scaling because advisory locks prevent overlap

Use advisory lock keys like:
hashtext('dfk:indexer:' + realm) and hashtext('dfk:finalizer:' + realm)

JOB 1: LISTING INDEXER (every ~10–30 min)
Steps:
1) Acquire advisory lock.
2) Read previous open auction IDs from DB:
   SELECT auction_id FROM tavern_heroes WHERE realm=$1 AND is_active=true;
3) Query GraphQL for current open auctions with pagination (first=1000, skip increments).
4) Build a Set currentOpenAuctionIds.
5) In a DB transaction:
   a) Mark all existing listings inactive for this realm:
      UPDATE tavern_heroes SET is_active=false WHERE realm=$1;
   b) Upsert current open auctions into tavern_heroes:
      - key: (realm, auction_id)
      - set is_active=true, last_seen_at=now(), last_indexed_at=now()
      - update hero fields, price fields, etc.
      - store hero_id_normalized from tokenId.normalizedId
      - store hero_id_padded from tokenId.id
      - store stat_genes/visual_genes/shiny
      - keep listed_at stable: if row exists, do not overwrite listed_at unless null.
6) Persist the current open auction IDs snapshot reference:
   Option A: store a “last_seen_index_run_at” timestamp and use tavern_heroes rows to infer prev vs current next time
   Option B: store open auctions into a small `open_auctions_snapshot` table keyed by (realm, run_id)
   Choose the simplest that stays DB-backed and idempotent.
7) Release lock.

JOB 2: AUCTION FINALIZER (every ~10–30 min, or triggered after indexer)
Purpose: finalize auctions that closed since last run.

Steps:
1) Acquire advisory lock.
2) Determine closedAuctionIds as:
   prevOpenAuctionIds - currentOpenAuctionIds
   IMPORTANT: because webapps cannot depend on memory, compute prev/current using DB-backed snapshots (from step C6).
3) For each closed auction_id:
   - Query GraphQL: saleAuction(id)
   - If purchasePrice != null:
       status='SOLD'
       purchase_price_wei = purchasePrice
       ended_at = endedAt
     else:
       status='DELISTED'
       purchase_price_wei = null
       ended_at = endedAt (if exists) else now()
   - Insert into tavern_sales with UPSERT on (realm, auction_id).
4) Snapshot hero traits “as-of listing time”:
   Copy hero fields into tavern_sales from the latest tavern_heroes row that had that auction_id
   (avoid relying on tokenId fields from saleAuction, which can reflect current hero state).
5) Release lock.

JOB 3: BACKFILL SOLD AUCTIONS (daily or on-demand)
This catches fast sales between snapshots.
- Query saleAuctions where purchasePrice_not: null and endedAt within last 90 days (or configurable).
- For each result, UPSERT into tavern_sales with status='SOLD', purchase_price_wei, ended_at, seller/winner.
- If hero snapshot fields are missing, fill best-effort from hero query.
