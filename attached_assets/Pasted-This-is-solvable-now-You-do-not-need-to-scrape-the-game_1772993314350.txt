The main correction to Replit’s model is this: there is not one global bonusType -> effect enum. The meaning of a code depends on the source contract/slot. DFK splits equipment across WeaponCore, ArmorCore, and AccessoryCore, and AccessoryCore itself serves both Accessory and Offhand slots; its equipmentType is 1 = Accessory, 2 = Shield, 3 = Focus. DFK also exposes all three equipment contracts through the Community GraphQL API, so you can wire this without bundle scraping. 

That means your sample 2 / 35 / 18 is only meaningful after you know the source:

ArmorCore: 2 = Spell Block chance, 18 = Fear resistance, 35 = Riposte. 

AccessoryCore (Accessory table): 2 = Magical Accuracy, 18 = Poison resistance, 35 = SPELL%. 

WeaponCore: 2 = Bleed on hit, 18 = Taunt on hit, 35 = Retaliate on physical damage. 

AccessoryCore (Offhand table): 2 = Block damage reduction; 18 and 35 do not exist there.

Two extra gotchas are likely behind the remaining mismatch. First, P.ACC/M.ACC are not only bonus-code driven: weapon data also has base physical accuracy fields accuracyRequirement + pAccuracyAtRequirement, and magic accuracy fields focusRequirement + mAccuracyAtRequirement, both with one-decimal precision. So if Replit only added equipment bonus codes but ignored those weapon fields, accuracy will still drift. 

Second, armor codes 36–53 are not normal additive stats. The docs literally say “Double the bonuses above this bonus…”, so the safe implementation is to process armor bonuses in slot order and treat those as prefix doublers, not as flat multipliers on the whole item. The docs do not spell out the exact comparator for X, but since they also say X = bonusScalar and equipment identity is keyed by type + displayId, my best read is that bonusScalar should be checked against the equipped item’s relevant displayId within that category/type. I would implement it that way and validate against this hero first.
