Yes — you were right. I was missing the **pet combat bonus layer**, and it materially changes the model.

Each pet carries exactly one encoded combat bonus in `combatBonus` plus its magnitude in `combatBonusScalar`. The raw ID encodes both the base bonus and the star tier, and the numeric bonus value is `combatBonusScalar / 10,000`. In your note, `1–79` means 1-star, `80–158` means 2-star with `base = rawId - 79`, and `160+` means 3-star with `base = rawId - 159`.

That means pet bonuses belong in the **static pre-bout calculation layer**, not just in live combat FX. Your note says pets can directly affect `BLK`, `SBLK`, `REC`, `P.DEF`, `M.DEF`, `CSC`, `EVA`, `SPEED`, `CHC`, `P.ACC`, `M.ACC`, `SER`, `P.RED`, and `M.RED`, depending on the base code. It also says some pet bonuses only modify one specific status resistance and **do not** raise global `SER`.

So the corrected stack is:

```ts
displayedStat =
  workbookBaseFormula
  + equipmentStaticFields
  + equipmentBonusCodes
  + heroPassiveTraits
  + petCombatBonus
  + liveBattleFX;
```

The pet decoder should look like this:

```ts
type DecodedPetBonus = {
  stars: 0 | 1 | 2 | 3;
  baseCode: number;
  bonusValue: number; // fraction, e.g. 0.001 = 0.1%
};

export function decodePetCombatBonus(
  combatBonus: number,
  combatBonusScalar: number
): DecodedPetBonus {
  if (!combatBonus) return { stars: 0, baseCode: 0, bonusValue: 0 };

  if (combatBonus >= 160) {
    return {
      stars: 3,
      baseCode: combatBonus - 159,
      bonusValue: combatBonusScalar / 10_000,
    };
  }
  if (combatBonus >= 80) {
    return {
      stars: 2,
      baseCode: combatBonus - 79,
      bonusValue: combatBonusScalar / 10_000,
    };
  }
  return {
    stars: 1,
    baseCode: combatBonus,
    bonusValue: combatBonusScalar / 10_000,
  };
}
```

And the pet application layer should be something like:

This revises a few of my earlier conclusions.

For the **wizard**, the `P.RED` / `M.RED` bug still stands. Your pet note says `Brick Wall` and `Null Field` are additive percent modifiers to `P.RED` and `M.RED`, not the core DEF-to-reduction transform. So the exact “card value = 2 × app value” relationship still points to your app using the wrong denominator in the DEF→RED conversion. Pet bonuses can add on top, but they do not explain that exact halving bug.

For **EVA**, I was too confident that the missing piece was only armor EVA. Your pet note shows `Slippery` adds directly to total EVA, so a pet can absolutely account for part of that recurring `~+5.x` EVA delta. The corrected interpretation is: some of the EVA gap may be armor, some may be pet, and some may still be live combat FX.

For the **status resistance panel**, the pet note is critical. It explicitly separates **global SER** from **one-off status resist pets** like `Outspoken` (Silence), `Lucid` (Confuse), `Moist` (Burn), `Insulated` (Chill), and so on. That means your app should not do `SER += petSpecificStatusResist`. Those belong only on the individual status lines.

That also gives a useful constraint on the beige tournament cards: since each pet has one encoded combat bonus, one pet can explain **one primary pet bonus type**, not a whole cluster of unrelated status spikes. So when the wizard card shows several elevated resist lines at once, pet bonuses are only part of the explanation; passives and live bout FX still have to be in the model.

For **speed**, the note says `Blur` is special: `petSpeedBonus = round(bonusValue × profileSpeed)`. It even gives an example where a 3-star Blur at scalar 10 still rounds to zero. So pet speed usually will **not** fix a 1-point speed mismatch unless the scalar is meaningfully larger.

For **P.DEF / M.DEF pct pets**, your note says `Heavy Hide`, `Magical Shell`, and `Omni Shell` apply percentage scaling to the defense base, and `Omni Shell` applies to both P.DEF and M.DEF. So these should be modeled as scaling on `(rawDef + scalingDef)`, not as flat DEF additions.

The other important correction is that many pet codes are **not numeric stat bonuses at all**. Your note lists a large set of proc-style effects — including `Threaten`, `Hobble`, `Shock`, `Petrify`, `Flow State`, `Rebalance`, `Guardian Shell`, and others — and says they should not be added to tracked numeric vitals as simple stat multipliers. It also flags `Rebalance (70)` as still unknown. So if one of the bout pets has a proc-only code, trying to force it into BLK/SER/EVA math will create false positives.

So the corrected formulas in your app should be:

```ts

sblk = sblkBase + equipSblk + traitSblk + petSblk + liveSblk;

pDef = pDefBase
     + pDefPctSourcesAppliedToBase // equip + pet Heavy Hide / Omni Shell + live pct buffs
     + flatPDefSources;

     + mDefPctSourcesAppliedToBase // equip + pet Magical Shell / Omni Shell + live pct buffs
     + flatMDefSources;

pRed = pDef / (50 * avgBattleLevel) + equipPRed + petBrickWall + livePRed;
```

* it does **not** replace the `P.RED/M.RED` denominator fix;
* it does **not** justify folding one-off status resist pets into `SER`;
* and it does **not** turn proc-only pet effects into numeric stat bonuses.
