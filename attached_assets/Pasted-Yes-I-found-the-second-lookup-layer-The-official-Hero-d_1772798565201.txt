Yes — I found the second lookup layer.

The official Hero docs show that `Passive1`, `Passive2`, `Active1`, and `Active2` are all part of the hero stat-gene layout, and then give a shared `Active/Passive Ability` code table: `0..7` map to `Basic1..Basic8`, `16..19` map to `Advanced1..Advanced4`, `24..25` map to `Elite1..Elite2`, and `28` maps to `Exalted1`. ([DFK Developer Docs][1])

The official combat docs then resolve those tier labels into actual trait names, but **separately for Active Traits and Passive Traits**. In other words, `Basic1` is overloaded: as an active trait it is **Poisoned Blade**, and as a passive trait it is **Duelist**. ([DeFi Kingdoms][2])

So for accessory bonus `36: "Gain Ability: X"`, the safest model is a **two-step** lookup:
`bonusScalar -> tierSlot -> activeName OR passiveName`. The accessory docs do confirm that bonus `36` is `Gain Ability: X`, and the gameplay docs say equipment can grant “special passive and active abilities,” but I did **not** find an explicit sentence saying bonus 36 always uses the hero gene ability-code table, nor one saying how to distinguish active vs passive from the accessory docs alone. That part is still an inference. ([DFK Developer Docs][3])

One practical gotcha: the Hero docs write the codes as `Basic1`, `Advanced1`, etc., while the combat docs write them as `Basic 1`, `Advanced 1`, etc. Normalize by removing spaces before joining the tables. ([DFK Developer Docs][1])

What I would implement right now is:

```ts
export function decodeAccessoryAbility(code: number) {
  return {
    code,
    slot: ABILITY_CODE_TO_SLOT[code] ?? null,
    active: ACTIVE_TRAIT_BY_CODE[code]?.name ?? null,
    passive: PASSIVE_TRAIT_BY_CODE[code]?.name ?? null,
    resolved: ACTIVE_TRAIT_BY_CODE[code] || PASSIVE_TRAIT_BY_CODE[code] ? true : false,
  };
}
```

That keeps the ambiguity explicit instead of forcing a wrong single-name map.

As a sanity check, if your live `bonusScalar` values for accessory bonus `36` mostly fall in `{0..7,16..19,24,25,28}`, that is strong evidence this is the correct namespace. That’s an inference from the official code table, not a directly stated DFK rule. ([DFK Developer Docs][1])
