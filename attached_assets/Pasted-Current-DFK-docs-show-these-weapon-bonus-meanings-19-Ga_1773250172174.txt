Current DFK docs show these weapon bonus meanings:

* `19` = **Gain X% chance to inflict Daze on basic attack when targeting a channeling enemy**
* `28` = **Increase Magical Damage dealt and reduce Healing Potency by X% each**
* `29` = **Increase Healing Potency and reduce Magical Damage dealt by X% each**
* `30` = **Decrease Physical and Magical Defense by X% each**

So `19` is not an unknown REC-type stat bonus; it reads as a conditional combat proc. `28/29/30` are compound bonuses that need to fan out into multiple fields instead of a single `EquipmentBonuses` key. ([Defi Kingdoms Developer Docs][1])

That means the Replit fix is:

* treat `19` as a **special effect/proc**, not a numeric REC bonus
* treat `28/29/30` as **multi-stat handlers**
* stop silently dropping unmapped codes

A minimal patch is to keep your existing flat `WEAPON_BONUS` map for simple cases, and add a `WEAPON_BONUS_SPECIAL` override for these four codes:

```ts
// Rename these fields to whatever your local canonical names are.
type EquipmentEffect =
  | { kind: "proc"; key: "dazeVsChannelingOnBasic"; chancePct: number };

type SpecialWeaponBonusHandler = (args: {
  x: number; // normalized scalar, using your existing scalar conversion
  out: EquipmentBonuses;
  effects: EquipmentEffect[];
}) => void;

const WEAPON_BONUS_SPECIAL: Record<number, SpecialWeaponBonusHandler> = {
  19: ({ x, effects }) => {
    effects.push({
      kind: "proc",
      key: "dazeVsChannelingOnBasic",
      chancePct: x,
    });
  },

  28: ({ x, out }) => {
    out.magicDamagePct = (out.magicDamagePct ?? 0) + x;
    out.healPotencyPct = (out.healPotencyPct ?? 0) - x;
  },

  29: ({ x, out }) => {
    out.healPotencyPct = (out.healPotencyPct ?? 0) + x;
    out.magicDamagePct = (out.magicDamagePct ?? 0) - x;
  },

  30: ({ x, out }) => {
    out.physDefPct = (out.physDefPct ?? 0) - x;
    out.magicDefPct = (out.magicDefPct ?? 0) - x;
  },
};

function applyWeaponBonus(
  code: number,
  rawScalar: number,
  out: EquipmentBonuses,
  effects: EquipmentEffect[],
) {
  const x = normalizeBonusScalar(rawScalar); // use your existing conversion

  const special = WEAPON_BONUS_SPECIAL[code];
  if (special) {
    special({ x, out, effects });
    return;
  }

  // existing simple single-field mapping path
  const key = WEAPON_BONUS[code];
  if (!key) {
    throw new Error(`Unhandled weapon bonus code ${code}`);
  }

  applyFlatBonus(key, x, out); // whatever your current simple path is
}
```

If your UI is only showing static rolled stats, `19` should go into an `effects` list rather than the main numeric summary. Forcing it into REC would be the wrong model.

Also, if the REC gap is still there after fixing this, the current armor table explicitly puts **Recovery chance** on armor bonus `41`, so the remaining delta is more likely in armor handling, passives, pets, or another non-weapon layer. ([Defi Kingdoms Developer Docs][2])
