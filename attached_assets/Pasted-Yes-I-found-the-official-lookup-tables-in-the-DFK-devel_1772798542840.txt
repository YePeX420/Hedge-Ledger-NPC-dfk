Yes — I found the official lookup tables in the DFK developer docs. The big takeaway is that there is **not one global `bonusId` namespace**. DFK publishes separate bonus mappings for **weapons**, **armor**, **accessories**, and **offhands**. Accessories and offhands live in the same AccessoryCore contract, but they still use different bonus tables, so your lookup key should be `(family, bonusId)`, not just `bonusId`. ([DeFi Kingdoms Developer Docs][1])

So your example of **bonus ID 7** depends on the family:

* **weapon[7]** = daze-on-hit proc chance
* **armor[7]** = speed %
* **accessory[7]** = status-effect resistance %
* **offhand[7]** = flat M.DEF ([DeFi Kingdoms Developer Docs][2])

For parsing, the docs show that weapons carry `bonus1..bonus4` plus `bonusScalar1..4`, while armor and accessory/offhand items carry `bonus1..bonus5` plus `bonusScalar1..5`. The docs also say accessory bonuses that use both `X` and `Y` pack them into one `uint16` scalar with `X = bonusScalar & 255` and `Y = bonusScalar >> 8`. ([DeFi Kingdoms Developer Docs][2])

That normalized object is derived from the official DFK accessory/offhand, armor, and weapon bonus tables. ([DeFi Kingdoms Developer Docs][3])

The special cases to preserve in your schema are:

* `accessory[36]` is **not self-describing**; it says “gain ability: X”, so `X` needs a second ability lookup table. ([DeFi Kingdoms Developer Docs][3])
* `accessory[231]` and `accessory[234]` use **packed X/Y** values, so you need the bitwise decode rule instead of treating the scalar as a single number. ([DeFi Kingdoms Developer Docs][4])
* `armor[36..53]` are **conditional synergy multipliers** that say “double the bonuses above when equipped with X …”; I would store the raw `X` parameter until you confirm exactly what secondary identifier it refers to. The docs clearly define the effect family, but the snippet I found does not further decode that `X` field. ([DeFi Kingdoms Developer Docs][5])

Most importantly, I do **not** see a direct “STR boost” style bonus ID in these official bonus tables. The bonus IDs are mostly combat procs, resistances, block/accuracy/damage modifiers, duel modifiers, and conditional synergies. Raw combat contribution from equipment also comes from other fields like weapon base damage / potency, speed modifier, armor defenses / evasion, and other non-bonus item stats. ([DeFi Kingdoms Developer Docs][2])

The clean implementation path is to seed a lookup table with columns like:
`family, bonus_id, key, scalar_mode, needs_secondary_lookup`.

That will let your Metis snapshot layer decode bonuses correctly without ever assuming `bonusId` is global.
