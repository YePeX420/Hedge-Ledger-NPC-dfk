The expanded object confirms the live game has a real tournament registry model in memory, and it is richer than the public community GraphQL API. The public GraphQL docs still only cover Heroes, Hero Sales/Rentals, Equipment, Pets, Pet Sales, Profiles, and Bazaar, while the published PvP contract docs expose battle-level interfaces on the Metis PvP Diamond, not a documented tournament interface. 

What your object tells us immediately:

It is almost certainly coming from an internal API/indexer or unpublished tournament facet wrapper, not a raw public subgraph response.

It already contains the three core layers you need:

tournament header fields

player_entries

bouts


The bouts[*].battle_id field is the most important bridge, because the public PvP Diamond does expose getBattle(...) and battle events on Metis, so this likely gives you a join from tournament data into the documented battle layer.

entry_period_start = 1741446000

tournament_start_time = 1741449600


That is a difference of 3600 seconds, so for this tournament the entry window is 1 hour.

That makes the card logic much clearer:

"Entries Close In" on the card is very likely tournament_start_time - now


I would still treat that as a practical inference until you compare it against one live accepting-entry card.

The object also shows the tournament source is not just a thin copy of the public battle settings struct. The public PvP docs currently show only one mustIncludeClass1 / includedClass1 slot in battle settings, but your tournament object has included_class_1, included_class_2, included_class_3 and matching must_include_class_* flags. That strongly suggests a separate tournament model or an unpublished extension beyond the public battle ABI. 

It also lines up with shipped feature history. The roadmap says DFK shipped Tournaments in February 2025, Hosted Tournaments in May 2025, shot-clock and sudden-death tournament updates in March and July 2025, and Hero Trait Tournament Restrictions in June 2025. Fields like shot_clock_duration, banked_shot_clock_time, max_hero_trait_score, max_team_trait_score, and max_trait_tier fit that timeline.

The best technical next step is to verify whether battle_id is the same namespace as the public PvP Diamond. The public docs expose getBattle(uint256) on 0xc7681698B14a2381d9f1eD69FC3D27F33965b53B, so test one of the values from bouts[*].battle_id, not bout_id. If getBattle(100000000027) returns a valid battle, your indexer path becomes:

tournament source -> bouts[].battle_id -> public PvP battle/events -> hero/equipment snapshots. 

Use this exact smoke test:

import { ethers } from "ethers";
import abi from "./PVPDiamond.json";

const provider = new ethers.JsonRpcProvider("https://andromeda.metis.io/?owner=1088");
const pvp = new ethers.Contract(
  "0xc7681698B14a2381d9f1eD69FC3D27F33965b53B",
  abi,
  provider
);

const battleId = 100000000027n; // from bouts[*].battle_id
const battle = await pvp.getBattle(battleId);
console.log(battle);

If that works, you do not need to wait for a tournament ABI to build most of the data layer.

If it does not work, then the tournament source is likely maintaining its own battle-id namespace and you’ll need the request URL or the create/join tx path to identify the contract/service behind it.
