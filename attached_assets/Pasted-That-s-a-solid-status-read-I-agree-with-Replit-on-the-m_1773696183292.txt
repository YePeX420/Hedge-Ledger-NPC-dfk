# What I would do next

## Priority 1 — Add a diagnostic mode before broader testing

This should be the immediate next task.

You want the extension to expose a small debug panel or console logger showing:

* current page recognized as combat page: yes/no
* hunt/session id detected: value or null
* hero cards found: count
* enemy cards found: count
* legal action buttons found: count
* consumables found: count
* battle log observer firing: yes/no
* latest parsed turn snapshot
* parse confidence score
* top 3 recommended actions
* selector matches by category

That will make first-session testing massively more useful.

Without that, testers will just say:

> “It doesn’t work”

and you won’t know whether the problem was:

* selectors
* SPA routing
* timing
* hidden shadow DOM
* iframe context
* stale React re-render timing
* missing hunt ID
* bad action parsing

---

## Priority 2 — Make selector strategy more resilient

Right now, selectors like:

```js
[class*="skill-btn"]
[class*="SkillBtn"]
```

are too brittle on a React production app.

Replit should move to a **tiered selector strategy**:

### Tier A

Stable semantic hooks if available:

* `data-*`
* `aria-label`
* button text
* tooltip text
* icon alt/title

### Tier B

Pattern-based selectors:

* partial class matching
* subtree matching
* nearby label matching

### Tier C

Fallback DOM inference:

* detect clickable buttons inside action panel region
* classify by visible text / icon / order

For DFK specifically, visible text and layout inference may be more reliable than hashed class names.

---

## Priority 3 — Fix session / hunt identification

If DFK is SPA-routed, the extension should not rely mainly on URL.

Better options:

* inspect DOM for visible hunt/battle title
* inspect page text for encounter names
* use mutation observer to detect route-state changes
* capture a combat-session fingerprint from:

  * encounter name
  * timestamp
  * enemy names
  * hero names
  * first parsed turn event

If true hunt ID is unavailable, generate a temporary synthetic session key like:

```text
encounterName + heroTeamHash + enemyTeamHash + startTimestamp
```

That is good enough for local testing.

You want:

* simple install instructions
* version label
* changelog
* debug mode on by default for early testers

That way testers can give actionable feedback.
